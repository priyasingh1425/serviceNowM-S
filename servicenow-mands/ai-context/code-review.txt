Enterprise Architecture Review – US-0000034
Branch: feature/US-0000034 (salesforce-copado)
Review Date: 2026-03-26
Scope: ~1,900 lines added across 46 files – Bi-directional Salesforce/ServiceNow integration
via Mulesoft for City of Minneapolis

Verdict: Pull Not Approved
This changeset is a solid start at a bi-directional ServiceNow integration, and the overall
approach — Queueable for outbound callouts, REST API for inbound, Mulesoft as middleware
— is sound. However, there are several areas that need to be addressed before this can move
forward to UAT. The most critical concern is security — the inbound API lacks input validation
and exposes internal implementation details on failure. Beyond that, there are bugs in bulk
safety and data integrity that need attention — these are not issues that UAT will catch, and
deploying them to a shared environment risks impacting other automation and integrations in
the org.

1.Security

Security No Input Validation on Inbound API
CityIntegrationAPI deserializes inbound JSON and passes values like
payload.IncidentRecID and payload.ParentRecID directly into SOQL queries without
validating they are well-formed 18-character Salesforce IDs. Examples:
processResolveIncident — WHERE Id = :payload.IncidentRecID
processCreateWorkNote — WHERE Id = :parentId
processReopenIncident — WHERE Id = :payload.IncidentRecID
Bind variables prevent SOQL injection, but malformed IDs (e.g., "abc123" ) will throw an
unhandled System.StringException that returns a 500 with a raw Apex stack trace to the
caller, leaking internal implementation details.
Security Unnecessary Global Access Modifier on MPLRequest
CityIntegrationAPI is a @RestResource class declared global , and MPLResponse is its
return type — so MPLResponse being global is required. However, MPLRequest is also

declared global even though it is only used internally for JSON.deserialize() within
CityIntegrationAPI . It should be public . Deploying it as global unnecessarily exposes
the class across namespace boundaries and cannot be downgraded once in production.
Recommendation Action-Level Authorization
CityIntegrationAPI routes all inbound requests through a single @RestResource endpoint
with an apiAction query parameter. Any authenticated API user (via Connected App) can call
any action — createIncident , resolveIncident , reopenIncident , createChangeRequest .
If the Connected App and Permission Set are properly scoped, this is acceptable. However, if
multiple external systems share the same Connected App, consider adding action-level checks
to limit which callers can invoke which actions.

Security Request Payloads Logged to Debug Logs
CityIntegrationAPI.handlePost() and handlePatch() both call System.debug('Request
Body: '+ body) unconditionally. Debug logs are accessible to any admin with the “Manage
Users” permission and can be retained for up to 24 hours. If request payloads contain PII or
sensitive customer data, this creates an exposure. This should be gated behind a custom
setting flag or removed.

2. Error Handling & Resilience

Bug Silent Failures on Outbound Callouts
Every method in API_ChangeRequestService and API_IncidentService catches Exception
and returns null . The callers in CreateRequestQueueable never check the return value. If a
Mulesoft callout fails, the failure is logged to System.debug() only — no record is created, no
retry is attempted, no one is notified.
Specific examples:

// API_ChangeRequestService.cls
} catch (Exception e) {
ApiResponseHelper.logException(LOG_PREFIX, 'createCR', e);
return null; // caller never checks this
}

java

This means if Mulesoft returns a 500, a timeout, or a network error, the Case update completes
successfully in Salesforce but the corresponding ServiceNow record is never created/updated.
The systems silently fall out of sync.
At minimum, the queueable should check return values and log failures to API_Response__c —
the logging object already exists and is used on the happy path.
Recommendation Retry Mechanism
There is no retry logic for transient HTTP failures (timeouts, 503 Service Unavailable, rate
limiting). Once a callout fails, that data is lost. A full retry / dead letter queue is a significant
architectural pattern and may not be expected for an initial build, but it should be on the
roadmap for production readiness.
Best Practice Broad Exception Catching
Every outbound service method catches Exception (the base class) rather than specific
exception types like CalloutException or JSONException . This suppresses programming
errors ( NullPointerException , TypeException ) that indicate bugs in the code, making them
invisible in production.
Recommendation Circuit Breaker
If Mulesoft is unavailable, every Case/Task/FeedItem update for the City of Minneapolis
account will enqueue a CreateRequestQueueable that makes a callout and fails. There is no
mechanism to detect repeated failures and stop enqueuing callouts until the downstream
system recovers. Under high Case volume, this could exhaust the org’s async job limits. A
circuit breaker (e.g., a Custom Setting flag that disables callouts when Mulesoft is known to be
down) would mitigate this.

WhatIt Should Do
Filtering should happen at the handler level before any queueable is enqueued. The handler
should check the Account/Record Type of the Cases and exit early for non-applicable records.
HelperCaseTrigger.validateRequiredFields is the one place in this changeset that does this
correctly — it checks Record Type and Account before doing any work. The same pattern
should be applied to ChangeRequestHandler .
Impact on Bulk Operations
Any bulk operation that touches Cases will be affected:
Data migrations/loads — bulk inserts or updates will enqueue unnecessary queueables
for every batch of 200
Batch Apex jobs — any batch that updates Case status or priority will compete for
queueable job limits
Flow/Process Builder bulk updates — scheduled actions or mass updates will trigger the
integration for every record
Data loader operations — routine maintenance updates will fire the handler on all records
In an org with significant Case volume, this pattern consumes async job capacity that other
integrations and automation depend on.

4. Governor Limit Risks

Best Practice Multiple Queueables Per Trigger Execution
ChangeRequestHandler.afterUpdate can enqueue up to 3 separate Queueable jobs in a
single trigger execution:
1. Status update cases
2. Priority update cases

3. One or more FeedItem sync jobs
The org-wide limit is 50 queueable jobs per transaction, shared with all other automation. Only
the FeedItem path checks Limits.getLimitQueueableJobs() — the status and priority paths
do not.
Best Practice Unbounded FeedItem Query
ChangeRequestHandler.afterUpdate queries all matching FeedItems with no LIMIT clause:

A Case with a large Chatter history could return thousands of records, hitting heap size limits.
Best Practice Fragile Queueable Chaining
CreateRequestQueueable.processFeedItems processes one FeedItem per execution and
chains to a new queueable for the rest. If any link in the chain throws an unhandled exception,
all remaining FeedItems in that chain are lost with no way to recover them.

5.Architectural Concerns

Best Practice Bypasses Existing Trigger Framework
The org has an established trigger handler framework already in use. This changeset ignores it
entirely and wires ChangeRequestHandler directly into CaseTrigger.trigger via static
method calls. This creates two competing trigger patterns in the same org, making it unclear
which pattern future developers should follow. Salesforce best practice is one trigger per
object delegating to a handler framework — mixing patterns increases maintenance complexity
and makes trigger execution order harder to reason about.
Best Practice Change Tasks Modeled as Child Cases
The commented-out code in CityIntegrationAPI.processCreateChangeRequest shows this
was originally built using standard Task objects, then pivoted to child Cases. Both parent

Change Requests and child Change Tasks use the same Change_Request record type. This
means:
No way to distinguish a Change Request from a Change Task by record type
Change Tasks appear in Case reports, dashboards, and list views alongside real Cases
Case assignment rules, escalation rules, and sharing rules apply to Change Tasks
unintentionally
Case metrics (open case count, average resolution time) are inflated
Recommendation Hardcoded to a Single Customer
CityIntegrationConstants hardcodes ACCOUNT_NAME = 'City of Minneapolis' and
CONTACT_NAME = 'City Of Minneapolis Service Desk' . Every query in the integration filters
by these literal strings:
CityIntegrationAPI.handlePost
CreateRequestQueueable.execute
HelperCaseTrigger.validateRequiredFields
If this integration will only ever serve one customer, constants are pragmatic. If the account is
renamed, merged, or if a second customer requires ServiceNow integration, all of these
queries break. Custom Metadata Types would make this configurable without code changes.

6. Naming Conventions

Recommendation No Domain-Based Naming
All functionality in this changeset flows from the Case object — the Case trigger fires the
handler, which enqueues the queueable, which calls the services. The class names don’t
reflect this shared domain. They use inconsistent prefixes, generic names, and unexplained
acronyms.
Inconsistent “API” Casing
The changeset uses three different conventions for the same concept:
API_ChangeRequestService — screaming case prefix with underscore
ApiResponseHelper — PascalCase, no underscore
CityIntegrationAPI — screaming case suffix
Generic Names That Will Collide
Class Risk
ErrorMessageBuilder Only handles DML Database.Error formatting for this integration, but the name implies an org-wide utility.

The next team will either create a duplicate or add unrelated logic here.

TriggerBypass Contains a single static boolean for FeedItem bypass, but the name implies a universal trigger bypass
framework. Next developer adds their bypass flag here instead of scoping it to their feature.

HttpRequestResponseModel Contains ServiceNow-specific DTOs ( CRCreateRequest , CreateIncidents , ResolveIncident ) but the name

gives no indication of what system or integration it belongs to.

Unexplained Acronyms
MPLRequest and MPLResponse — “MPL” is never defined anywhere in the code or comments.
Future developers must read the class internals to understand what these represent.
Recommended: Domain-First Naming

Since everything originates from Case, use the pattern {Domain}{Purpose}{Pattern} :

Case and could legitimately be reused by other integrations.
Benefits

A new developer can identify the full surface area without opening any files
No ambiguous prefixes or acronyms that require tribal knowledge
Self-documenting — the class name tells you what object it touches and what system it
integrates with

8. Testing

Zero Test Classes
There are no test classes in this changeset. For ~1,900 lines of new Apex including triggers, a
REST API, queueables, and HTTP callouts, this means:
No HttpCalloutMock implementations for the 8 distinct Mulesoft callout methods
No test coverage for the 5 inbound REST API actions
No negative test cases (malformed IDs, missing required fields, duplicate records,
Mulesoft errors)
No bulk trigger tests (200+ record inserts/updates)
If the intent is to defer test coverage until code is finalized for UAT — the concern is that many
of the issues identified in this review (Section 3: Bulk & Batch Safety, Section 9: Race
Conditions) are behavioral defects that will not surface in manual UAT testing. They only
appear under bulk data volumes, concurrent operations, or specific trigger re-entry
sequences. Writing tests that simulate these conditions is the only reliable way to validate the
code behaves correctly before it reaches production. Deferring tests until after the code is
“finalized” risks finalizing code that has fundamental issues baked in — issues that are
significantly harder and more expensive to fix after the fact than during active development.

Salesforce requires 75% code coverage for production deployment, but the real value of tests
here is validating behavior, not hitting a coverage number.
Untestable Design
CityIntegrationAPI performs SOQL, DML, business logic, and response formatting inline in
~490 lines. Writing meaningful tests against this class requires setting up the full data graph
(Account, Contact, Case, Queue, Record Types, FeedItems, Approval_Participants__c) for
every test method. Extracting business logic into testable service classes would significantly
reduce test setup complexity and improve coverage quality. If tests are being deferred because
the current design makes them too difficult to write, that is a signal the design needs to
change.

9. Data Integrity

Bug No Idempotency on Incident Creation
CityIntegrationAPI.processCreateIncident creates a new Case on every call with no
duplicate checking. If Mulesoft retries a failed request (e.g., timeout where the Case was
actually created), a duplicate Case is inserted. In contrast, processCreateChangeRequest does
check for existing records via Customer_Rec_Id__c — this same pattern should be applied to
incidents.
Bug Race Condition Between APIResponseTrigger and CaseTrigger
The following sequence creates a race condition:
1. Case is created/updated, ChangeRequestHandler fires CreateRequestQueueable
2. Queueable calls Mulesoft, gets back a CHG/INC number
3. ApiResponseHelper logs an API_Response__c record
4. APIResponseTrigger fires and stamps Customer_Rec_Id__c on the Case
5. This Case update re-fires CaseTrigger -> ChangeRequestHandler.afterUpdate
6. ChangeRequestHandler sees the new Customer_Rec_Id__c and queries all existing
FeedItems to sync as work notes
Result: every historical Chatter post on the Case gets sent to ServiceNow as a work note the
moment the CHG/INC number is assigned. This is almost certainly unintended — only new
FeedItems should sync.

Recommendation Transactional Integrity on Outbound
If a single Case update triggers both a status change and a priority change,
ChangeRequestHandler.afterUpdate enqueues two independent queueables. Each makes its
own callout to Mulesoft. If the status update callout succeeds but the priority update fails,
ServiceNow reflects the new status but the old priority, with no mechanism to detect or
reconcile the inconsistency. This is a hard problem in distributed systems — flagging it for
awareness, not necessarily expecting it to be solved in v1.

10. Observability & Operations

Best Practice Exception Logging Bypasses API_Response__c
The codebase has an API_APIResponse_INS class that logs to an API_Response__c custom
object, and it’s used on the happy path. However, when an exception occurs in the outbound
service classes, ApiResponseHelper.logException only writes to System.debug() . This
means successful callouts are tracked in API_Response__c but failures are only visible in
debug logs, which expire after 24 hours and require active monitoring to capture.
Recommendation Alerting on Integration Failures
There are no Platform Events, email alerts, custom notifications, or Slack integrations for
callout failures. The only way to discover the integration is broken is:
A user manually notices that a ServiceNow ticket wasn’t created
An admin happens to be watching debug logs at the time of failure
Someone queries API_Response__c and notices gaps (but failures aren’t logged there)
A full alerting infrastructure may not be expected for an initial build, but at minimum, logging
failures to API_Response__c (which already exists) would make them discoverable.

11. Deployment Risk

Best Practice Flow Changes
Participant_Change_Request_Process.flow-meta.xml has structural changes to an approval
flow that interacts with the Approval_Participants__c object — the same object that
CityIntegrationAPI.processUpdateChangeRequestApproval modifies via DML. Flow and Apex

code modifying the same object in the same transaction can cause unexpected trigger re-
entry or conflicting field updates. This interaction needs explicit testing.

12. Mulesoft Considerations

Recommendation HTTP 202 Treated as Failure
ApiResponseHelper.processApiResponse only treats HTTP 200 and 201 as success:

Mulesoft commonly returns HTTP 202 (Accepted) for asynchronous processing — meaning
Mulesoft accepted the request and will forward it to ServiceNow, but hasn’t completed yet.
With the current code, a 202 from Mulesoft would be treated as a failure: no response object
returned, and the API_Response__c log would record isSuccess = 'false' . This should be
verified against the actual Mulesoft API contract.
Recommendation Callout Timeout at Maximum
MulesoftApiClient sets the HTTP timeout to the Salesforce maximum of 120,000ms (2
minutes):

If Mulesoft is degraded but not fully down, each callout will block for up to 2 minutes before
timing out. In CreateRequestQueueable , multiple cases are processed sequentially in a single
execute — one slow Mulesoft response blocks all remaining cases in that queueable
execution. This is common practice and may be fine depending on expected volumes, but
worth noting.
